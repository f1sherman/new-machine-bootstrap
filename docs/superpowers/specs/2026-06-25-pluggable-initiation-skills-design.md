# Pluggable initiation-skill identifiers and superpowers-plugin opt-out

**Date:** 2026-06-25
**Status:** Approved (design)

## Problem

nmb hardcodes one specific skill library (`superpowers`) in two ways:

1. It installs the `superpowers@claude-plugins-official` Claude plugin
   unconditionally (`roles/common/tasks/main.yml`).
2. The two "initiation" design skills it cares about —
   `superpowers:brainstorming` and `superpowers:systematic-debugging` — are
   referenced literally in the base `CLAUDE.md` fragment and in several
   behavioral hooks (Claude and Codex).

A downstream provisioner that layers on top of nmb may want to use a
**different skill library** that provides equivalent initiation skills under
different names. Today that is impossible without forking nmb: the plugin is
force-installed, and every hook matches the `superpowers` skill ids verbatim.

Note that an alternate library may rename the skills entirely (different
namespace **and** different verb), so a namespace-prefix swap is insufficient —
the full skill identifiers must be configurable.

## Goals

- Let a downstream provisioner opt out of nmb installing the superpowers Claude
  plugin, without removing the default behavior for everyone else.
- Let a downstream provisioner override the two initiation-skill identifiers so
  that the base `CLAUDE.md` guidance **and** all behavioral hooks (Claude +
  Codex) recognize the alternate skills.
- Keep the defaults exactly as they are today (`superpowers:brainstorming`,
  `superpowers:systematic-debugging`, plugin installed) so a plain nmb run is
  unchanged.

## Non-goals

- The Codex skill clone path (`~/.codex/superpowers`, pinned in
  `vars/tool_versions.yml`, linked into `~/.agents/skills`) is **out of scope**;
  it is unaffected.
- No backwards-compatibility detection or dual-matching: the override is one
  explicit value, not a heuristic that accepts both old and new ids.

## Override channel

A downstream provisioner invokes nmb's `bin/provision` as a child process, so
the override must cross a process boundary. nmb already reads environment
variables via `lookup('env', ...)`, so **environment variables are the override
channel**. Each new variable has a default equal to today's behavior, so an
unset environment is a no-op.

| Env var | Default | Effect |
| --- | --- | --- |
| `NMB_INSTALL_SUPERPOWERS_PLUGIN` | `true` | When false, skip installing `superpowers@claude-plugins-official`. |
| `NMB_BRAINSTORMING_SKILL` | `superpowers:brainstorming` | Identifier for the brainstorming/design initiation skill. |
| `NMB_DEBUGGING_SKILL` | `superpowers:systematic-debugging` | Identifier for the debugging initiation skill. |

## Design

### Part A — Plugin install opt-out

The Claude plugin install loop currently installs a fixed list including
`superpowers@claude-plugins-official`. Introduce a boolean fact
`install_superpowers_plugin`, derived from `NMB_INSTALL_SUPERPOWERS_PLUGIN`
(default `true`). Build the plugin list so the superpowers entry is included
only when the flag is true; `pr-review-toolkit` is always installed. The
install task is unchanged except for the now-conditional list.

When the flag is false, nmb neither installs nor uninstalls superpowers — it
simply does not manage it. Removing an already-installed copy is the downstream
consumer's responsibility (it has the context to do so safely).

### Part B — Pluggable initiation-skill identifiers

Two role variables, sourced from the env vars above with superpowers defaults:

```yaml
brainstorming_skill: "{{ lookup('env', 'NMB_BRAINSTORMING_SKILL') | default('superpowers:brainstorming', true) }}"
debugging_skill:     "{{ lookup('env', 'NMB_DEBUGGING_SKILL')     | default('superpowers:systematic-debugging', true) }}"
```

The skill ids are consumed by static shell scripts (hooks) and one Markdown
fragment. Rather than convert every script from `files/` to `templates/`, nmb
writes a single generated, sourced config file and the scripts read from it.
This keeps the override in one place and minimizes churn to working hooks.

**Generated config** — nmb templates a small file (e.g.
`~/.claude/hooks/initiation-skills.sh`) that exports the configured ids in the
forms the hooks need:

```sh
# generated by nmb — do not edit
NMB_BRAINSTORMING_SKILL='superpowers:brainstorming'
NMB_DEBUGGING_SKILL='superpowers:systematic-debugging'
# bare verbs (portion after the ':') for prompt-text matching
NMB_BRAINSTORMING_VERB='brainstorming'
NMB_DEBUGGING_VERB='systematic-debugging'
```

The bare-verb forms are derived from the full ids (text after the last `:`) so
a single override drives both the namespaced `case` matches and the
prompt-text regexes.

**Consumers updated to source the config and build their patterns from it:**

- `CLAUDE.md.d/00-base.md` → becomes a template; the tmux-subject line
  references `{{ brainstorming_skill }}` / `{{ debugging_skill }}`.
- `claude/hooks/block-initiation-skill-on-main.sh` — `case` arm built from the
  two full ids (keeps the literal `_spec-first`/`_spec-to-pr`/`_fix` arms).
- `claude/hooks/remind-agent-subject-on-skill.sh` — `case` arm from the two
  full ids.
- `claude/hooks/remind-repo-start-on-dev-prompt.sh` — regex built from the two
  bare verbs (keeps literal `_fix`/`_spec-first`/`_spec-to-pr`).
- `bin/codex-remind-agent-subject-on-prompt` — regex from the two full ids.
- `bin/codex-remind-repo-start-on-dev-prompt` — regex from the two bare verbs.

`bin/_recover-agent-sessions` strips a literal leading ``Using `brainstorming`.``
prefix from session summaries; this is cosmetic display cleanup. It will read
the bare brainstorming verb from the generated config so the strip stays correct
under an override.

**Deployment ordering:** the generated config must be written before/with the
hooks and bin scripts so the scripts can source it at runtime (it is read at
hook-execution time, not provision time, so strict task ordering only matters
for first-run presence). nmb writes it into both `~/.claude/hooks/` (for Claude
hooks) and ensures the bin scripts can locate it (same path, sourced by
absolute `~/.claude/hooks/initiation-skills.sh`).

### Testing

- `remind-repo-start-on-dev-prompt.sh.test` already exercises the repo-start
  reminder. Extend it to run twice: once with the default config (superpowers
  verbs) and once with an override config (alternate verbs), asserting the
  reminder fires for the configured skills and stays silent for the others.
- Add a focused test that the `case`-based hooks
  (`block-initiation-skill-on-main.sh`, `remind-agent-subject-on-skill.sh`)
  match the configured full ids and ignore unrelated skills, under both default
  and override configs.
- End-to-end: `bin/provision --check` / `--diff` with the env unset (no change
  to deployed ids) and with the env set (deployed config + assembled CLAUDE.md
  reflect the overrides).

## Downstream consumer (abstract)

A private downstream provisioner that runs nmb as an upstream step will:

1. Export `NMB_INSTALL_SUPERPOWERS_PLUGIN=false` and the two skill-id overrides
   around its invocation of nmb's `bin/provision`.
2. Remove the now-unmanaged superpowers plugin and install its preferred skill
   library + marketplace via its own playbook.

Those concrete steps live in the consumer's own (private) spec, not here.

## Risks

- The behavioral hooks gate the user's core workflow (repo-start prompts,
  tmux-subject prompts, block-on-main). A regression silently weakens those
  reminders. Mitigation: TDD on the hook tests under both default and override
  configs, plus end-to-end verification that an unset environment produces no
  diff against current behavior.
